# nju-se-thesis-self-check

V 0.1 

南京大学软件学院、智能软件与工程学院专业学位硕士论文送审前学生自检 Agent Skill（通过这份自检不代表评委一定会让你通过）。

Skill 基于学院论文要求，以及上百份实际的论文评审意见综合而成。

学生提供论文 PDF，skill 自动抽取论文题目并请学生确认，判定「工程型 / 学术型」，按对应类别的 51 条 checklist（含 5 条红线）逐项审查，给出带原文页码定位的反馈、红线高亮、评议结论估计，最后写出一份 Markdown 报告。

> **免责声明**：本项目基于规则匹配，不能替代真人盲审。真实评议结论取决于盲审专家的主观判断、所在批次整体水平，以及本 skill 没覆盖的维度（代码质量、Demo 可运行性等）。仅供学生自查参考。

---

> 流程末尾内置一个**可选**步骤：用 https://github.com/op7418/Humanizer-zh 做 AI 文风检查 + 错别字检查。论文较长，本 skill 会先询问是否进行；同意后会按章节切成多块逐块检查，并把结果**追加**到 `<论文题目>-review.md` 的第七节。本部分仅供参考。需先安装 humanizer-zh：`npx skills add https://github.com/op7418/Humanizer-zh.git`。

## 是什么

一个 skill（应该可以用在大部分 Agent 上）。在用户明确说出触发词（如「论文自检」「送审前自查」）并给出 PDF 路径时调用本 skill，按论文类别走 51 条 checklist：

- **5 条红线**——评审会「一票否决」或「严重影响通过」的项
- **30 条学术质量**——研究方法、文献综述、实验设计等
- **11 条结构内容**——章节安排、逻辑闭环
- **5 条参考文献问题**——基础规范、质量门槛、引用格式、来源严肃性

输出报告写到 `<论文同目录>/<论文题目>-review.md`，对话里只贴摘要。

## 适用对象

- 南京大学软件学院、智能软件与工程学院专业学位硕士（**软件工程领域工程硕士**）
- 送审前自查阶段
- 工程型（系统 / 平台 / 工具实现）和学术型（方法 / 模型 / 实验）论文均可

非软院、非专硕、非本校用的 checklist 可能不完全匹配。

## 安装

在 Agent 中说：

安装一个 skill： https://github.com/dongshao/nju-se-thesis-self-check 



需要安装 Python 依赖（可以使用 Agent 来安装）：

```bash
pip install pdfplumber
```

或者重启 Agent会话，skill 即被自动加载。

## 使用

在 Agent 中直接说：

```
论文自检 /path/to/your-thesis.pdf
```

跳过类别判定（已知是工程型 / 学术型）：

```
论文自检 /path/to/your-thesis.pdf --type=engineering
论文自检 /path/to/your-thesis.pdf --type=academic
```

全自动模式（不询问任何确认，从步骤 1 一气呵成跑到步骤 8）：

```
论文自检 /path/to/your-thesis.pdf --auto
全自动论文自检 /path/to/your-thesis.pdf
自动模式 论文自检 /path/to/your-thesis.pdf
```

自动模式触发词：消息中含「全自动」「自动模式」「auto」「--auto」「-y」任一即可。题目抽取、类别判定、是否做文风检查这三处 Gate 全部由 skill 自己的判断直接通过；判定依据写入报告。**缺省（不带 `--auto`）保留人工确认**。

完整触发词列表：「论文自检」「论文自查」「送审前自查」「我的论文有什么问题」「我的论文能过吗」「self-check」「pre-defense check」。

## 流程

1. **抽骨架 + 提取题目**——从 PDF 摘要页 / 封面正则抽取真实论文题目（不依赖文件名 / PDF metadata，因为 NJU 模板的 metadata 标题往往是文件 ID）
2. **类别判定 Gate**——通过章节关键词信号判别工程型 / 学术型，请学生确认
3. **加载对应 51 条**checklist
4. **红线扫描**——先跑 5 条红线
5. **全量审查**——剩余 46 条，每条按「评判 + 定位 + 修改建议」模式
6. **生成报告**到 `<论文同目录>/<论文题目>-review.md`
7. **对话输出摘要**——红线项数、警告数、预估评议结论
8. **（可选）文风与错别字检查**——询问后再运行；调用 humanizer-zh 按章切块逐块扫描 24 类 AI 写作痕迹，同时做错别字 / 标点 / 全半角 / 术语一致性检查，结果追加到报告第七节

## 文件结构

```
nju-se-thesis-self-check/
├── SKILL.md                          # skill 主文档（被 Agent 自动加载）
├── references/
│   ├── engineering-50.md             # 工程型 51 条 checklist
│   └── academic-50.md                # 学术型 51 条 checklist
└── scripts/
    └── extract_skeleton.py           # PDF 骨架抽取脚本
```

每条 checklist 含 `id / 分区 / 是否红线 / 问题 / 检查方法 / 失败模板 / 修改建议` 七个字段。

## 评议结论估计

按以下顺序判定（取首个匹配）：

| 条件 | 估计 |
|---|---|
| 红线 ≥ 1 | **不合格** 风险高 |
| 警告 ≥ 8 | **修改后答辩** |
| 警告 4–7 | **合格** |
| 警告 ≤ 3 且 通过 ≥ 45 | **良好** 或更高 |

仅作参考，不构成正式评议。

## 已知限制

- 仅做基于文本骨架的规则匹配，**无法判断**：代码是否真的能跑、Demo 是否真实、实验数据是否伪造（除非数字明显可疑）、图表是否抄袭。
- 查重需自行去 CNKI 知网。本 skill 只能粗略提示「该段表述与公开资料雷同概率高」。
- 骨架抽取依赖「第X章」中文章节标识。完全英文论文 / 章节命名不规范的论文会触发「未识别章节」退化策略，审查准确率会下降。
