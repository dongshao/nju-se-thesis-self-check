# 工程型论文自检 51 条

适用对象：软件产品研发类论文（系统 / 平台 / 工具 / 模块的设计与实现）。

格式说明：
- 每条含 `id / 分区 / 是否红线 / 问题 / 检查方法 / 失败模板 / 修改建议`
- 检查方法里 `<skeleton>` 指步骤 1 生成的骨架 md 文件路径
- 检查方法里 `<pdf>` 指原始 PDF 路径
- 失败模板里 `{X}` 是 Claude 应在审查时填入的具体内容

## 🔴 红线（5 条）

### ENG-RED-1 — 学术规范：研究行为
- **问题**：是否存在抄袭、剽窃、伪造数据/图表/结论的迹象
- **检查方法**：
  1. 抽查第二章中 3 段长描述性文字（每段 100+ 字），用 google/bing 检索原文；高度雷同则警示
  2. 检查第四、五章的截图与界面元素是否前后矛盾（如同一系统不同章节用了完全不同的 UI 风格）
  3. 检查实验数据是否合理（精度 100%、改进幅度 >50% 等可疑数字）
- **失败模板**：第二章 P.{页码} "{原句}" 与公开资料 "{来源}" 高度相似（>70% 字面重合）。江苏省评议细则把"抄袭、剽窃"列为学术规范单项不合格，整篇论文将被一票否决。
- **修改建议**：用自己的话彻底重写该段，确保 CNKI 知网查重不命中；若必须保留原文表述，必须加引用标注。

### ENG-RED-2 — 文不对题（标题党）
- **问题**：标题中的核心关键词，能否在第三、四章正文找到对应的功能实现
- **检查方法**：
  1. 取标题 → 切成 3-5 个名词短语（如"智能 / 营销 / 预算 / 管理 / 系统"）
  2. 排除"系统/平台/设计与实现"这类通用词
  3. 剩余每个关键词 grep `<skeleton>` 全文，统计在第三章及以后的命中次数
  4. 命中次数 = 0 或仅出现在标题/摘要中：标题党
- **失败模板**：标题中的"{关键词}"在第三章及以后正文中仅出现 {n} 次（grep 结果），未见对应功能模块或实现章节。这是评审"一票否决"的常见情形。
- **修改建议**：二选一：(1) 在正文补充与该关键词对应的实质性内容（不是只补一段定义）；(2) 修改论文标题，去除该关键词，使标题与正文实际工作严格匹配。

### ENG-RED-3 — 正文篇幅
- **问题**：第一章至末尾章（不含目录、致谢、参考文献、附录）总页数 ≥ 60 页
- **检查方法**：从骨架元信息取 `第一章 page_start` 与 `末尾章 page_end`，计算差值
- **失败模板**：正文实际篇幅约 {X} 页，未达 NJU 软院 60 页底线。
- **修改建议**：补充第三、四章（设计与实现）的细节，特别是模块的接口、关键数据流、关键算法、异常处理等。**禁止**通过堆砌代码、放大图、增加废话来凑页数——评审专家一眼能识别。

### ENG-RED-4 — 研发工作量饱满度
- **问题**：设计与实现章节合计 ≥ 40 页（实际论文章节切分各异——有的把"需求/架构/详细设计与实现"分散在第三、四、五章，有的把"系统测试"另起一章。本检查按"研发工作量"实质判定）
- **检查方法**：
  1. 识别哪些是"设计与实现"章（典型为：需求分析 / 体系结构 / 详细设计与实现 / 系统测试，含编号即可）
  2. 把这些章的总页数加起来
  3. 合计 ≥ 40 页 → ✅；合计 < 40 页 → 红线
- **失败模板**：设计与实现相关章节（{识别到的章节}）合计 {X} 页，少于 40 页。这通常被评审视为"研发工作量不饱满"。
- **修改建议**：检查是否有核心模块的设计或实现描述被简化掉了。补充：(1) 关键模块的详细数据流；(2) 关键接口的设计与调用链；(3) 关键算法的伪代码；(4) 系统测试用例及结果。

### ENG-RED-5 — 致谢/正文遗留模板占位
- **问题**：是否存在"致谢示例""TODO""XXX""[待补充]""待填""未定义书签"等遗留文本
- **检查方法**：`grep -nE "致谢示例|TODO|XXX|待补充|待填|未定义书签|占位文本|示例文字" <skeleton>`
- **失败模板**：在 P.{页码} 发现遗留模板/占位文本："{原文}"。这种疏漏直接暴露论文未经过最后一遍通读，会让评审认为研究态度不端正。
- **修改建议**：全文 grep 上述关键词清理；同时全文搜索 `[1]` `[2]` 之外的方括号引用占位、`图 X.Y` 这类未替换的图编号占位。

## 学术质量（30 条）

### ENG-A-01 — 摘要双向一致性（评审高频考点）
- **问题**：摘要中提到的功能/问题，是否能在正文找到对应论述；正文重点功能是否在摘要被概括
- **检查方法**：
  1. 从摘要提取 5-8 个关键名词（如"链路追踪 / 流式响应 / Span 扩展"）
  2. 每个关键词 grep `<skeleton>` 全文，统计在第三章及以后的命中次数
  3. 命中 0 次的关键词标记为"摘要-正文断层"
- **失败模板**：摘要中提到"{X}"，但全文 grep 仅命中 {n} 次，且未在任何章节深入论述。
- **修改建议**：二选一：(1) 摘要删除该关键词；(2) 在第三、四章对应位置补充论述并统一术语。

### ENG-A-02 — 摘要四要素完整性
- **问题**：中文摘要是否完整说明"为什么做（动机）+ 怎么做（方法）+ 带来什么效果（结果）+ 解决什么问题（贡献）"
- **检查方法**：读摘要全文，逐项核对 4 个要素
- **失败模板**：摘要主要论述"{X}"，但缺少 {缺失的要素}。摘要是评审最仔细读的部分，门面性极强。
- **修改建议**：摘要长度 3/4 页以上，建议结构：(1) 1-2 句行业背景与痛点；(2) 1-2 句指出现有方案不足；(3) 3-4 句概括本文系统的核心模块及技术路线；(4) 2-3 句给出量化的研发效果。

### ENG-A-03 — 第一章问题动机的具体性
- **问题**：第一章是否给出**具体场景与痛点**，而非泛泛"随着信息化发展"
- **检查方法**：
  1. 读第一章前 3 页
  2. 搜索是否含"具体业务场景""量化指标""现有方案的具体不足"等
  3. 若全章节只能找到行业泛泛描述，标记警告
- **失败模板**：第一章"研究背景"以行业宏观叙述为主（P.{页码}），未给出具体业务场景下的真实痛点（如"X 业务在 Y 场景下，因 Z 导致 N 个问题"）。
- **修改建议**：在第 1.1 节加一段具体场景描述（最好是真实项目中遇到的），用 1-2 个量化数字（如"日均请求 100 万次"、"平均故障定位耗时 4 小时"）锚定问题严重性。

### ENG-A-04 — 第二章技术-后文一致性（评审高频致命项）
- **问题**：第二章列出的所有技术，是否在第三、四章的设计与实现中被实际使用
- **检查方法**：
  1. 提取第二章二/三级标题的技术关键词
  2. 每个关键词 grep `<skeleton>` 第三章及以后的命中数
  3. 命中数 = 0 → 该技术在后文未使用
- **失败模板**：第二章 §{节号} 介绍了"{技术}"（P.{页码}），但全文 grep 在第三章及以后命中 0 次。
- **修改建议**：二选一：(1) 在第三/四章补充该技术的具体应用论述（不只是"我们使用了 X"，而是说明 X 解决了哪个具体问题、如何配置、关键代码示意）；(2) 从第二章删除该技术的描述及对应小节。

### ENG-A-05 — 第二章每个技术的"作用说明"
- **问题**：第二章每介绍一个技术，是否在该小节开头或末尾说明"该技术在本文项目中发挥什么作用、如何应用"
- **检查方法**：抽查第二章前 3 个技术小节，看是否有 1-2 段"在本文中…"的承接说明
- **失败模板**：第二章 §{节号}"{技术}"小节为单纯的技术介绍（类似教科书章节），缺少"在本文项目中如何使用"的承接说明。
- **修改建议**：在每个技术小节末尾补 1 段（3-5 行）："本文使用 {X} 实现 {具体模块}，主要利用其 {具体特性} 来解决 {具体问题}。"

### ENG-A-06 — 第二章避免低水平/陈旧技术堆砌
- **问题**：第二章是否避免"Java 语言""J2EE""HTML5""MySQL 数据库基础"等低水平/陈旧技术介绍
- **检查方法**：grep 第二章是否含"Java 语言简介""J2EE 概述""HTML5 介绍""MySQL 简介"等
- **失败模板**：第二章 §{节号} 介绍 "{技术}"（P.{页码}），属于本科级或常识级技术，会被评审认为综述水平不足。
- **修改建议**：删除此类小节。第二章应聚焦本系统使用到的**有技术含量**的框架/方法/算法（如分布式追踪标准、容器编排、向量数据库、特定 LLM 工程组件等）。

### ENG-A-07 — 第二章篇幅控制
- **问题**：第二章篇幅是否 ≤ 8 页（除非整篇 > 85 页可适当增加）
- **检查方法**：从骨架算第二章页数；若整篇 > 85 页则放宽阈值至 12 页
- **失败模板**：第二章 {X} 页，超过推荐的 8 页（整篇 {Y} 页）。
- **修改建议**：删除非必要的技术泛泛介绍，每个技术控制在 1 页内；删除显著与本文工作无关的内容。

### ENG-A-08 — 业界横向对比（评审高频考点）
- **问题**：是否在相关工作或总结章节，与同类业界系统/产品做了**横向对比**（不只是简单提及）
- **检查方法**：
  1. grep "对比""相比"等词在第二章和末尾章的命中段落
  2. 看是否有对比表，对比了至少 3 个维度（架构 / 功能 / 性能 / 部署 / 成本）
- **失败模板**：未发现与同类业界系统的横向对比。在大模型/可观测/支付/数据平台等热门方向，往往有 LangSmith / Langfuse / 蚂蚁 / 阿里 Dataworks 等成熟同类产品，缺少对比会让评审质疑工作差异化。
- **修改建议**：在第二章或末尾章添加一个对比表，明确指出本系统与 2-3 个同类系统在架构解耦、功能完整性、性能、部署难度等方面的差异与优势。

### ENG-A-09 — 创新点凝练
- **问题**：是否在绪论或总结明确**列点**说明本文的独立贡献（"本文的主要贡献：(1)... (2)... (3)..."）
- **检查方法**：grep 第一章和末尾章中"贡献""创新"等词附近段落
- **失败模板**：未在第一章或末尾章见到明确列点的贡献声明。
- **修改建议**：在第一章末尾补一节"本文主要工作"，列 3-5 条贡献，每条 1 句话，必须是**实际做出来的工作**而非空泛"提出了……"。

### ENG-A-10 — 创新点边界（避免概念过度拔高）
- **问题**：标题/摘要/创新点声明里是否过度使用"智能""先进""创新"等空词
- **检查方法**：
  1. 统计标题、摘要、第一章末尾的"智能""智慧""先进""创新性""前沿"出现次数
  2. 检查这些词是否对应了实际的算法模型 / 学习能力 / 自适应机制
- **失败模板**：标题/摘要使用"{词}"（如"智能预算管理"），但正文实现实际为规则触发或自动化核算（P.{页码}），缺少基于算法模型的智能能力。这类"概念拔高"在答辩时会被质疑。
- **修改建议**：二选一：(1) 在绪论或总结中明确界定本系统中"智能"的边界（如"流程自动化与数据驱动可视化"），避免误解；(2) 修改用词，更换为"自动化"等更贴近实际的表述。

### ENG-A-11 — 需求分析的真实性
- **问题**：第三章需求分析是否使用了用例图 + 用例文本，且来自实际业务场景而非编造
- **检查方法**：
  1. 用骨架定位第三章页范围（如 P.31-44），用 awk/sed 抽出第三章原文段
  2. 在**第三章范围内**（不要 grep 全文，否则第六章测试用例的"正常流程/异常流程"会误命中）grep "用例图""用例文本""actor""前置条件""主流程""异常流"
  3. 仅用例图但无用例文本（前置/主流程/异常流命中 0）→ ⚠️ 警告
  4. 用例图与用例文本均无 → ❌ 不合格
- **失败模板**：第三章 P.{页码} 有用例图（图 {X-Y} {图名}），但 grep "前置条件 / 主流程 / 异常流" 在第三章范围内命中 0 次，缺少标准用例文本。
- **修改建议**：在第三章功能性需求节，为最重要的 2-3 个用例补完整的用例文本（用例编号、参与角色、前置条件、主流程编号步骤、异常流、后置条件）。每个用例文本约半页篇幅。

### ENG-A-12 — 系统架构 4+1 视图（评审高频考点）
- **问题**：第三章是否参考"4+1"视图模型（逻辑/开发/进程/物理 + 用例）做架构描述
- **检查方法**：grep 第三章"逻辑视图""开发视图""进程视图""物理视图""部署视图""4+1"
- **失败模板**：第三章架构设计仅含一张总体结构图（P.{页码}），未按 4+1 视图模型展开。架构维度单一会被评审认为设计深度不足。
- **修改建议**：至少补充 3 个视图：(1) 逻辑视图（核心组件及关系）；(2) 开发视图（代码包/模块组织）；(3) 部署视图（容器/服务器/网络拓扑）。

### ENG-A-13 — 数据库设计完整性
- **问题**：信息系统类论文，是否提供 ER 图 + 数据库表结构
- **检查方法**：
  1. 判断是否信息系统类（grep "数据库""SQL""数据表""字段"在前五章命中数 ≥ 5）
  2. 若是 → 用扩展的 grep 模式检查 ER 图：`ER图|E-R图|实体关系图|实体-关系|实体之间.*关系`
  3. 检查图列表是否含 ER 图（grep "图 [0-9]+-[0-9]+ .*ER\|图 [0-9]+-[0-9]+ .*实体关系"）
  4. 检查表字段定义（grep "字段名|主键|外键|索引"）
  5. 若全文有 SQL → 抽 2 条 SQL，核对其涉及的表名/字段是否在表结构中定义
  6. ER 图、字段表、SQL 一致性，三者任一缺失 → ⚠️
- **失败模板**：本论文为信息系统类（含 {n} 个数据表表名描述），但 grep "ER 图 / E-R 图 / 实体关系图" 命中 {n} 次，未见标准 ER 图（图列表抽查也未发现）。
- **修改建议**：在第三或第四章数据库设计节补充：(1) ER 图（核心实体及关系，可用标准的"矩形=实体、菱形=关系、椭圆=属性"或 Crow's Foot 表示法）；(2) 关键表的字段表（字段名、类型、是否主键、是否索引、说明）；(3) 全文 SQL 涉及的表名/字段必须与表结构英文名一致。

### ENG-A-14 — 模块名贯穿一致性
- **问题**：摘要、目录、功能结构图、章节标题、本章小结中的模块名是否完全一致（不同别名混用会扰乱阅读）
- **检查方法**：
  1. 提取功能结构图旁的模块名（从图标题/正文上下文）
  2. 抽样这些模块名 → grep 摘要、目录、各章小结的命中
  3. 比对是否有"用户管理 / 用户中心 / 账户模块"等同义不同名情形
- **失败模板**：模块"{X}"在摘要中称为"{别名 1}"，在第三章节标题中称为"{别名 2}"，在功能结构图中称为"{别名 3}"。
- **修改建议**：全文统一一个标准称谓；二级模块名不要与一级模块同名，避免指代不清。

### ENG-A-15 — 功能结构总图与论文标题系统名称一致
- **问题**：功能结构总图的系统名称，是否与论文标题完全一致（标题改了之后忘改图是常见问题）
- **检查方法**：
  1. 取论文标题中的系统名（如"基于 X 的 Y 系统"中的 Y）
  2. 在第三章功能结构图所在页用 Read 取上下文，核对图标题
- **失败模板**：论文标题为"{标题}"，但第三章 P.{页码} 的功能结构图标题为"{图中名称}"，名称不一致。
- **修改建议**： 重新编辑该图，将系统名称改为与最新标题一致；同时检查所有章节出现的系统名称是否同步。

### ENG-A-16 — 系统测试章节
- **问题**：是否在第四章设置二级标题或单独一章写【系统测试】，且**结合系统业务实质**给出测试用例与结果，而非抄《软件测试》基础知识
- **检查方法**：grep "系统测试""测试用例""测试结果"在第四、五章的命中
- **失败模板**：未见独立的系统测试章节/小节；或测试章节充满"黑盒测试""白盒测试"基础概念介绍，缺少基于本系统真实业务的测试用例。

## 结构内容（10 条）

### ENG-S-01 — 目录三级层级 + 编号连续 + 无孤立三级
- **检查方法**：(1) 检查目录是否使用了四级编号（如 1.1.1.1）；(2) 检查每个二级章节下三级条目数 ≥ 2（避免孤立三级）；(3) 编号连续性
- **失败模板**：目录中 §{节号} 出现四级编号（应删去）；§{节号} 仅 1 个三级条目（孤立三级，应合并或拆出兄弟节）；§{节号} 编号 {n} 后直接跳到 {n+2}（不连续）。
- **修改建议**：合并四级到三级；为孤立三级增加兄弟节或将该三级提升为二级；重新编号确保连续。

### ENG-S-02 — 第一章末尾【本文组织结构】

- **检查方法**：grep 第一章末尾"组织结构""本文结构""章节安排"附近段落
- **失败模板**：第一章未见【本文组织结构】小节；或简介过长/过短。
- **修改建议**：补该小节，每章用 2-4 行概括，要言简意赅，必须列出该章重点论述的模块名/子系统名。

### ENG-S-03 — 每章必有"本章小结"

- **修改建议**：补本章小结。设计/实现章节的小结必须列出本章重点论述的模块名。

### ENG-S-04 — 章/节标题体现项目名/模块名

- **检查方法**：检查第三、四章标题（一级和二三级）是否含项目名或模块名

- **修改建议**：将通用标题如"系统设计"改为"{项目名}{模块功能}的设计"，让评审一眼看出每章在做什么。

- **检查方法**：从骨架取摘要页起止；统计字符数；检查是否含具体模块名
- **失败模板**：中文摘要篇幅 {X} 字（约 {Y} 页），少于建议的 3/4 页；或仅含套话，未列出具体模块/子系统名。

- **失败模板**：英文摘要段落数 ({X}) 与中文摘要 ({Y}) 不一致；或英文摘要遗漏中文摘要 §{n} 的内容。

- **修改建议**：替换为真实姓名与职称；个人感慨可在此抒发，但避免空洞煽情。如当前是盲审版本，到正式归档时记得补上真实致谢。

## 格式严重问题（6 条）

### ENG-F-01 — 参考文献基础规范
- **问题**：参考文献 ≥ 40 篇 + 每篇正文都有引用 + 引用-文献双向匹配
- **检查方法**：(1) 数文献条目；(2) 抽查 10个引用编号 → 看在文献列表是否存在；(3) 抽查 10 个文献 → grep 是否在正文有引用
- **失败模板**：参考文献 {n} 篇（要求 ≥ 40）；引用 [{X}] 在文献列表中未定义；文献 [{Y}] 未在正文出现引用。
- **修改建议**：补足文献至 40 篇；删除孤儿引用或补充对应文献；删除未引用的文献条目。

### ENG-F-02 — 参考文献质量门槛
- **问题**：近五年文献占比 ≥ 1/3；学位论文引用 ≤ 3 篇；纯编程书/网文 ≤ 3 篇；arXiv 预印本引用 ≤ 5 篇（预印本未经同行评议，过多会拉低参考文献整体严肃性）；不引用学科评估 B- 以下高校的学位论文
- **检查方法**：抽样统计文献年份分布、类型分布；用 `grep -inE "arxiv\.org\|arXiv:[0-9]{4}\." <skeleton>` 在参考文献区段统计 arXiv 条目数
- **失败模板**：近五年文献占比 {X}%（要求 ≥ 33%）；学位论文引用 {n} 篇（≤3）；编程类网文 {n} 篇（≤ 3）；arXiv 预印本 {n} 篇（建议 ≤ 5，>5 给出警告）。
- **修改建议**：替换陈旧文献为近 1-3 年的代表性论文（顶会、顶刊、知名工业界报告）；删除超额的学位论文引用与编程网文；超过 5 篇的 arXiv 预印本优先替换为已正式发表在顶会/顶刊的版本（多数 arXiv 论文最终会进 ICLR / NeurIPS / ACL / ICSE 等会议，引用正式版本更稳妥）。

### ENG-F-03 — 英文文献门槛 + 防伪英文文献
- **问题**：英文原版文献 ≥ 10 篇；不要把中文期刊误写为英文（如《计算机技术与发展》写成 *Computer Technology and Development*）
- **检查方法**：抽查英文文献 → 在 DBLP/ACM/IEEE 检索条目是否存在；中英文期刊名是否对应正确
- **失败模板**：英文文献 {n} 篇（< 10）；文献 [{X}] 标注为英文期刊 "{Y}"，但 DBLP 检索无该条目，疑似将中文期刊错写。
- **修改建议**：补足真实英文文献；将疑似错写的回复为中文标注（中文作者名、中文期刊名）。

### ENG-F-04 — 引用格式规范
- **问题**：(1) 引用使用 **[N] 数字编号**格式；(2) 中括号在句号内；(3) 多作者参考文献条目里用 et al. / 等
- **检查方法**：
  1. 抽查正文 10 处引用 → 应该是 [1] [12] [82] 等数字编号格式 ✅
  2. 若发现使用 [Smith et al., 2020] / [张三等, 2020] 等作者-年代格式 → ❌ 这是旧规范，需改为 [N] 数字编号
  3. `grep -E "[a-zA-Z]+(.+), [12][0-9]{3}\]"` 检测作者-年代误用（用 ERE：`-E`）
  4. `grep -E "。\[[0-9]+\]"` 检测中括号在句号外的情形
- **失败模板**：P.{页码} 使用 [作者, 年代] 格式（如 "{原文}"），应改为 [N] 数字编号格式；或 P.{页码} 引用 "{原句}" 中括号在句号外。
- **修改建议**：用 EndNote / Zotero 切换到数字编号样式（如 GB/T 7714-2015 数字格式或 IEEE 数字格式）；批量调整中括号位置到句号前；参考文献条目中的多作者用 et al. / 等。

### ENG-F-05 — 图表编号 + "如图 X.Y 所示"
- **问题**：图编号、表编号连续无重复；"如图所示""如下图所示"应改为"如图 X.Y 所示"
- **检查方法**：grep "如图所示""如下图所示""如下表所示""如表所示" 全文命中
- **失败模板**：发现 {n} 处"如图所示/如下图所示"未点名图编号；图 {x.y} 编号重复 / 缺失。
- **修改建议**：批量替换为"如图 X.Y 所示"；用 Word 自动编号工具重排图表编号。

### ENG-F-06 — 非同行评议来源处理
- **问题**：参考文献中是否混入行业咨询报告 / 厂商博客 / 网络博客 / GitHub 链接等非同行评议来源。这类材料若需引用，应处理为：
  (1) 移至正文脚注或括注（含访问日期）；
  (2) 或在参考文献条目中**明确标记**为"行业报告 / 厂商博客 / 在线资源 / 技术文档"以提示其非同行评议性质；
  (3) 优先替换为对应的同行评议论文 / 标准化文档（RFC / ISO / IEEE 标准 / W3C 规范等）
- **典型问题来源**：
  - 行业咨询报告：Gartner / IDC / Forrester / 中研普华 / 艾瑞咨询等的研究报告或新闻稿
  - 厂商博客 / 产品页：阿里云开发者社区、AWS 博客、Microsoft Tech Community、Vectra AI 等厂商博客或产品白皮书
  - 框架/库官网博客：vuejs.org/guide、reactjs.org/blog 等官方文档（除非引用规范 / API 标准段）
  - GitHub README 或 Issue 链接
  - 个人技术博客（CSDN、知乎专栏、Medium 等）
- **检查方法**：
  1. 抽查参考文献条目，识别 URL 类条目（grep `https?://|www\.` 在参考文献区段）
  2. 域名归类：`gartner.com / idc.com / forrester.com / chinairn.com / iresearch.com.cn` → 行业报告；`developer.aliyun.com / aws.amazon.com/blogs / cloudblogs.microsoft.com / *.vectra.ai` → 厂商博客；`vuejs.org / reactjs.org / nodejs.org` → 框架官网（区分规范段 vs 博客）；`github.com` → GitHub 链接；`csdn.net / zhihu.com / juejin.cn / medium.com` → 个人/社区博客
  3. 看条目是否已**明确标记**类型（如末尾有 `[行业报告]` `[厂商博客]` `[在线资源]` `[Z]` 在线电子资源参考文献类型标识）
  4. 未标记 → ⚠️
- **失败模板**：参考文献 [{N}] {标题}（{域名}）属于 {行业报告 / 厂商博客 / 在线社区文章 / GitHub 链接 / 厂商产品页}，作为正式学术参考文献的形式与正式性偏弱，但条目未做类型标记。
- **修改建议**：三选一处理：
  (1) **优选**：将该条目从参考文献删除，在正文对应处改为脚注或括注，含 URL + 访问日期，例如"……[Vue.js 官方性能指南，https://vuejs.org/guide/best-practices/performance.html，访问于 2026-05-01]"；
  (2) **次选**：替换为对应的同行评议论文 / 标准化文档（如把 Vue 官方博客换成 Vue 团队在 ICSE / OOPSLA 发的论文，把 Vectra 产品报告换成相关 USENIX Security 论文，把 Gartner 报告换成同主题的 SoTA 综述论文）；
  (3) **保留时**：在条目末尾加类型标记，如 `[行业报告]` `[厂商博客]` `[Z 在线电子资源]`，并加访问日期，让评审一眼看出非同行评议性质。
- **白名单（不算违规）**：
  - 标准化文档（RFC、ISO、IEEE Std、W3C Recommendation、GB/T）：天然权威，URL 是规范地址而非博客
  - 厂商官方技术白皮书 / 标准 SDK 文档（标注 `[R]` 报告或 `[S]` 标准类型）
  - 经典数据集网站（如 ImageNet、HuggingFace 数据集页）：作为数据来源引用
  - 论文官方代码仓库 GitHub 链接（仅当与同行评议论文条目并列出现时）
