# 软件工程图表规范性评估清单

适用对象：步骤 6（图表规范性检查）使用。本清单不进入红线计数；缺图/不规范一律计为本步骤产生的「警告」。

---

## 通用前置：图注与图编号

每张图都应满足以下基础规范，未满足者无论图类型一律记一条警告：

- **图注存在**：图下方有「图 X-Y 标题」或「图 X.Y 标题」格式的图注（X = 章号，Y = 图序）
- **章内连续编号**：图编号在所在章内从 1 起连续，不跳号
- **正文引用**：图至少在所在章正文有一处「如图 X-Y 所示」「图 X-Y 给出……」之类的显式引用，不能"放图不提"
- **清晰度**：模型看图能辨识主要文字和连接关系；糊到看不清算"图质量不达标"

检查方法：
- 图注用 `grep -nE "^图\s*[0-9]+[\-．.][0-9]+" <skeleton>`，统计每章图编号
- 正文引用用 `grep -nE "如图\s*X\-Y|图\s*X\-Y\s*所示|图\s*X\-Y\s*给出" <skeleton>`（X-Y 替换实际编号）
- 清晰度由模型直接读 `<论文同目录>/figs/page-NNN.png` 判定

## DGM-1 — 用例图（Use Case Diagram）

**期望出现位置**：工程型论文第三章「需求分析」。学术型可无。

**评估要点**（按 UML 2.x 标准）：

1. **Actor 表达**：参与者用人形 stickman 图标（或被框起来的 actor 关键字），不能直接画方框
2. **Use Case 表达**：用例用椭圆，椭圆内为动宾短语（如"提交订单"），不能用名词或单字
3. **System Boundary**：系统边界用矩形框圈住所有用例，框上方标系统名
4. **关系正确性**：
   - actor 与 use case 之间是实线（关联）
   - `<<include>>` 用虚线箭头，箭头指向被包含的子用例（强制复用）
   - `<<extend>>` 用虚线箭头，箭头指向被扩展的基用例（可选扩展），方向与 include 相反
   - 泛化用空心三角箭头（actor 间或 use case 间继承）
5. **Actor 命名**：用角色名（如"销售经理"），不要用具体人名或岗位 ID
6. **粒度合理**：单张用例图 use case 数量在 5-25 之间为佳；过少（< 3）说明系统简单到不必画，过多（> 30）说明该拆子图

**典型问题**：
- actor 画成方框 / 用例画成方框 → "图形元素不符合 UML"
- include / extend 箭头方向画反 → "关系语义颠倒"
- 没有 system boundary → "缺系统边界，分不清哪些用例属本系统"
- 用例命名为"用户管理"这种名词 → "用例名应为动宾短语"

**失败模板**：第 X 章图 X-Y「{标题}」存在 UML 规范问题：{具体问题列表}。建议参照 UML 2.x 用例图规范修正。

## DGM-2 — 4+1 架构视图

**期望出现位置**：工程型论文第三章或第四章「系统架构」。

**评估口径**（已与用户确认走"先 grep 关键词、未声明则跳过"）：

1. **触发条件**：用 `grep -nEi "4\+1|4 ?\+ ?1|Kruchten|逻辑视图|开发视图|进程视图|物理视图|场景视图" <skeleton>` 命中任一关键词才进入本项检查
2. **未命中**：直接跳过 DGM-2，不记警告（避免误伤未声明用 4+1 的论文）
3. **命中后**：检查论文里是否声明了"采用 4+1 视图模型"或等价表述
   - 仅含"4+1"字样但未明确说明用了 Kruchten 4+1：写一条提示，建议补一句"本文按 Kruchten 4+1 视图模型组织系统架构"
   - 已明确声明：检查正文是否分别给出五个视图（逻辑 / 开发 / 进程 / 物理 / 场景）的说明性内容（**不要求每个视图都有专门的图**，只要求正文提及并阐述）

**典型问题**：
- 声明了用 4+1 但只画了"系统总体架构图"一张，没有分视图论述 → "4+1 论述不完整"
- 把"前端 / 后端 / 数据库"三层硬套成"4+1" → "概念混用"

**失败模板**：第 X 章 P.{页码} 声明使用 4+1 架构视图，但仅给出 {Y} 个视图的论述，缺 {缺失视图列表}。建议补全或改用其他架构描述方法。

## DGM-3 — ER 图（实体-关系图）

**期望出现位置**：工程型论文第四章「数据库设计」或第三章「详细设计」。涉及关系数据库的系统应有。

**评估要点**：

1. **实体表达**：实体用矩形，矩形内为实体名（名词，如"用户"）
2. **属性表达**：属性用椭圆挂在实体上（经典 Chen 表示法），或在实体矩形内分行列出（紧凑表示法）。两种风格选一即可，不能在同一图里混用
3. **关系表达**：关系用菱形，菱形内为关系名（动词，如"拥有"），两端连接相关实体
4. **基数标注**：实体与关系连线两端必须标基数（1:1 / 1:N / M:N，或 0..1 / 1..* 等 UML 风格）。**没有基数标注是最常见的不规范**
5. **主键标识**：每个实体至少标出主键属性（带下划线或 PK 标记）
6. **弱实体**：依赖型实体用双框矩形，依赖关系用双框菱形（仅当论文确实有弱实体时检查）

**典型问题**：
- 关系画成线条+文字而非菱形 → "关系符号不规范"（容忍，但提一笔）
- 缺基数标注 → "无法判断实体间数量关系"
- 实体属性凌乱，多张图风格不统一（有的挂椭圆、有的内嵌） → "表示法不一致"

**失败模板**：第 X 章图 X-Y「{标题}」作为 ER 图存在以下问题：{具体问题}。建议补全基数标注与主键标识。

**注意**：很多论文用「数据库表结构图」代替 ER 图（直接画表+外键关系），这种是合法的设计阶段产物但不是严格意义的 ER 图。如果论文同时声称"ER 图"又画的是表结构图，记一条"图类型与命名不符"。

## DGM-4 — 类图（Class Diagram）

**期望出现位置**：工程型论文第四章「详细设计」。

**评估要点**（按 UML 2.x 标准）：

1. **类框三段式**：每个类用矩形分三栏（类名 / 属性 / 方法）。只画类名一栏的不算类图，是包图或简化示意图
2. **可见性符号**：属性和方法前应有 `+` (public) / `-` (private) / `#` (protected) / `~` (package)
3. **类型标注**：属性标 `name: Type`，方法标 `method(param: Type): ReturnType`
4. **关系符号**：
   - 关联（实线）
   - 聚合（空心菱形 + 实线，整体-部分但部分可独立存在）
   - 组合（实心菱形 + 实线，整体-部分且部分生命周期受整体控制）
   - 继承（空心三角箭头 + 实线）
   - 实现（空心三角箭头 + 虚线，类对接口）
   - 依赖（虚线箭头）
5. **重数标注**：关联线两端可标 1 / 0..1 / * / 1..* 等
6. **避免过度填充**：每张类图类数控制在 5-15 之间为佳；过多说明该按子系统拆图

**典型问题**：
- 缺可见性符号 → "未区分 public/private"
- 聚合 / 组合 / 关联 三种关系混用同一种线条 → "关系语义不清"
- 把对象图（具体实例）当类图 → "类图与对象图混淆"

**失败模板**：第 X 章图 X-Y「{标题}」作为类图存在以下问题：{具体问题}。

## DGM-5 — 流程图（Flowchart / 活动图）

**期望出现位置**：工程型论文第四章「详细设计」描述关键算法或业务流程时。学术型描述方法主流程时也常用。

**评估要点**：

1. **图形元素语义**（ANSI/ISO 5807）：
   - 椭圆（圆角矩形）：起止节点
   - 矩形：处理步骤
   - 菱形：判定/分支
   - 平行四边形：输入/输出
   - 圆形小节点：连接符（跨页或跨图）
2. **起止完整**：必有"开始"和"结束"节点；不能从矩形开头、悬空结尾
3. **判定分支标注**：菱形输出的两条线必须分别标"是 / 否"或"Y / N"，不能让读者猜
4. **方向一致**：自上而下或自左而右，不要乱绕
5. **如果是 UML 活动图**：
   - 起止节点用实心黑点（开始）和带圈黑点（结束）
   - 判定用菱形带 guard 条件 `[条件]`
   - 并发用 fork/join 黑粗线
   - 泳道用矩形分块标 actor

**典型问题**：
- 矩形/菱形混用（用菱形当处理步骤）→ "图元语义错"
- 判定分支没标 是/否 → "分支语义缺失"
- 没有结束节点 → "流程未闭合"

**失败模板**：第 X 章图 X-Y「{标题}」作为流程图存在以下问题：{具体问题}。

## DGM-6 — 时序图 / 部署图 / 状态图（出现才查）

**评估口径**：这三类图非工程型论文必须出现，**仅在论文里实际画了才查**。学术型同理。

简明检查点：

- **时序图**：必须有 lifeline（虚线垂直）、激活条（lifeline 上的细矩形）、消息箭头（实线同步 / 虚线异步 / 实心箭头返回）。常见错误：所有消息都画成实线箭头，看不出同步/异步/返回。
- **部署图**：节点用三维立方体，组件用矩形带左侧两个小方块。常见错误：部署图画成了"网络拓扑示意图"，没有 UML 节点符号。
- **状态图**：状态用圆角矩形，转移用箭头标 `event[guard]/action`。常见错误：转移上只写事件名，没有 guard 与 action。

**失败模板**：第 X 章图 X-Y 作为 {图类型} 存在以下问题：{具体问题}。

## 工程型论文的关键图缺失检查（仅工程型）

工程型论文应包含以下三类图至少**两类**。完全缺失则记一条本步骤的警告（不上红线）：

1. **用例图**：出现在需求分析章
2. **系统架构图**（任意形式：分层 / 微服务 / 模块依赖 等，不强求 4+1）：出现在体系结构章
3. **ER 图或表结构图**（涉及数据库的系统必须有）

检查方法：
- grep `<skeleton>` 全文找图注里包含"用例 / 架构 / 部署 / ER / 实体 / 表结构"等关键词的图
- 三类全缺 → 警告："缺系统结构性图表，论文工程描述以纯文字为主"
- 缺一类（如缺用例图）→ 警告："缺用例图，需求分析以纯文字描述会降低可读性"
- 缺二类 → 警告升级："缺多种关键工程图，建议补全"

**学术型论文不做此检查**：方法论文以方法框图、模型架构图、实验流程图为主，不强求 UML 体系。

## 模型不支持图像输入时的退化策略

如果步骤 6 的能力探针失败（模型不能 Read 图像），本步骤跳过所有视觉评估。但仍可做以下基于文本的检查：

1. 用 grep 统计每章图注数量，列出每章「图 X-Y 标题」清单
2. 检查图注是否符合「图 X-Y 标题」格式
3. 检查正文是否有"如图 X-Y 所示"的引用
4. 工程型论文的关键图缺失检查（基于图注关键词）仍可做

报告里写明"当前模型不支持图像输入，本节仅做基于图注文本的规范检查；UML 图形元素的规范性建议人工抽查"。
